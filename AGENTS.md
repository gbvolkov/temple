# Постоянные инструкции по работе с CMS

При исправлении, тестировании или планировании работ по недостаткам CMS необходимо сначала прочитать и затем соблюдать канонический план:

- `outputs/cms-shortcomings-remediation-plan.md`

По умолчанию этапы выполняются в указанной там последовательности. Сначала исправляется функциональность и автоматически очищаются данные; обязательное участие человека откладывается до этапа редакционной проверки.

Человека можно обязательно привлекать только для решения по содержанию конкретного материала, если решение нельзя достоверно получить из данных или формальных правил. Технические, программные, интерфейсные и тестовые решения не должны перекладываться на человека.

После каждого этапа обязательно выполнять применимые серверные, Playwright smoke, полные регрессионные и source-data тесты. Этап нельзя считать завершённым, пока обязательные тесты не прошли, известные `XFAIL` не сверены с реестром недостатков, а исходные runtime-данные не подтверждены как неизменённые.

Тестовый набор является живой частью функциональности CMS:

- при обнаружении нового функционала немедленно добавлять его в матрицу покрытия и создавать позитивные, негативные, ролевые и переходные тесты;
- при изменении поведения одновременно изменять соответствующие fixtures, тесты, ожидаемые результаты и документацию;
- не оставлять устаревшие тесты, фиктивные проверки покрытия или `XFAIL`, не соответствующие текущему поведению;
- при обнаружении нового дефекта сначала добавлять отдельный characterization-тест и запись в реестр, затем исправлять дефект;
- после расширения или изменения тестов повторно выполнять весь применимый регрессионный контур, а не только новый тест.

Отклоняться от плана или менять это правило можно только по явному указанию пользователя.
