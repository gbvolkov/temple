# Новый сайт и CMS храма святителя Иннокентия

В репозитории находятся адаптивный публичный сайт, рабочая CMS на FastAPI + SQLite, импорт read-only снимка старого MODX, дизайн-система и презентация.

## Быстрый локальный запуск

```powershell
uv sync
$env:CMS_ENV="development"
uv run uvicorn server.app:app --host 127.0.0.1 --port 4174
```

- сайт: `http://127.0.0.1:4174/`
- редактор: `http://127.0.0.1:4174/cms.html`
- локальный вход: `admin` / `temple-demo`

Демонстрационный пароль включается только при `CMS_ENV=development`. Для production скопируйте `.env.example` в `.env` и задайте длинный случайный `CMS_BOOTSTRAP_PASSWORD` до первого запуска.

## Импорт старого сайта

Скрипт читает только локальный JSON-снимок, создаёт черновики и не обращается к старому MODX на запись.

```powershell
uv run python -m server.importer --dry-run
uv run python -m server.importer
```

Повторный запуск идемпотентен: неизменившиеся записи пропускаются. Старые пути добавляются в таблицу 301-редиректов. Все импортированные страницы помечаются «Нужна проверка».

## Полный read-only crawl и зеркалирование файлов

Краулер использует Playwright, регулярно сохраняет очередь и продолжает с checkpoint после остановки:

```powershell
uv sync --group crawl
uv run playwright install chromium
uv run python -m server.legacy_crawl
```

Он исключает `/manager/`, документы и внешние домены из очереди страниц. Запросы выполняются только методом GET; старый MODX не изменяется.

После crawl создайте реестр и сначала выполните dry-run зеркалирования:

```powershell
uv run python -m server.legacy_inventory
uv run python -m server.media_mirror
uv run python -m server.media_mirror --execute
uv run python -m server.full_import --dry-run
uv run python -m server.full_import
uv run python -m server.media_report
uv run python -m server.migration_audit
```

Зеркало дедуплицирует файлы по SHA-256, ограничивает размер, разрешает только HTTPS старого домена и возобновляется по manifest. Импорт заменяет исходные URL локальными только для успешно зеркалированных файлов.

Текущие отчёты:

- `outputs/legacy-inventory.md` — классификация обследованных URL;
- `outputs/missing-legacy-media.md` и `.csv` — отсутствующие на исходном сервере файлы и связанные материалы;
- `outputs/migration-readiness.md` — контрольные барьеры перед переключением домена.

## Публикация и рабочие версии

CMS хранит рабочую и опубликованную ревизии раздельно. Сохранение изменений
опубликованного материала создаёт новый скрытый черновик; посетители продолжают
видеть прежнюю ревизию до повторной проверки и публикации. Рабочий процесс:

1. редактор сохраняет материал и отправляет его на проверку;
2. publisher или admin возвращает материал редактору, публикует сразу либо
   назначает будущую дату;
3. архив и корзина скрывают материал, а восстановление создаёт скрытый черновик;
4. история ревизий позволяет скопировать старую версию в новый черновик;
5. журнал действий не сохраняет полные тексты или секреты в metadata.

Планировщик запускается вместе с приложением, выполняет первый проход сразу и
повторяет проверку раз в минуту. Все mutation-запросы используют текущий номер
`version`; устаревшее состояние получает HTTP `409`.

## Baseline перед изменением схемы

Исходный код до завершения CMS закреплён аннотированным тегом
`baseline-before-completion-20260718`. Версионированные миграции можно
проверять без записи в рабочую БД:

```powershell
uv run python -m server.migrations status --database data/cms.sqlite3
uv run python -m server.migrations verify --database data/cms.sqlite3
uv run python -m server.migrations up --dry-run --database data/cms.sqlite3
```

На production сначала выполняется согласованное резервирование и restore
drill. Скрипт делает предварительный `rsync`, останавливает только контейнер
CMS, создаёт SQLite `.backup`, запускает сайт и затем проверяет архив в
отдельном временном контейнере:

```bash
cd ~/temple
bash scripts/production_baseline.sh
```

После успешного выполнения скрипт печатает `BACKUP_ID`, пути архива и отчёта,
SHA-256 и длительность остановки. Вторая копия скачивается на Windows вне
репозитория:

```powershell
powershell -ExecutionPolicy Bypass -File scripts/download_baseline.ps1 `
  -RemoteHost volkov@cheetan `
  -BackupId baseline-YYYYMMDDTHHMMSSZ
```

Содержимое `.env` не печатается и не попадает в git; оно хранится только в
архиве baseline с правами `600` и в локальном каталоге с ограниченными ACL.

После проверки SHA-256 второй копии на Windows пункт 1 применяется отдельной
командой. Перед рабочей БД новый образ сначала запускается на восстановленной
копии архива:

```bash
cd ~/temple
bash scripts/production_apply_point1.sh baseline-YYYYMMDDTHHMMSSZ
```

Перед production-развёртыванием этапа 2 создаётся ещё один свежий baseline.
Checkout резервной копии должен быть предком развёртываемого коммита, а схема
БД должна оставаться на версии 2. Архив обязательно скачивается на Windows и
там проверяется. Полученный SHA-256 передаётся rollout-скрипту; новый образ
сначала запускается на восстановленной копии БД на `127.0.0.1:18002`:

```bash
cd ~/temple
bash scripts/production_baseline.sh

# После download_baseline.ps1 и проверки локальной копии:
LOCAL_BACKUP_SHA256=<sha256-из-Windows> \
  bash scripts/production_apply_stage2.sh baseline-YYYYMMDDTHHMMSSZ
```

Rollout проверяет схему 3, все 1 757 материалов, единственную опубликованную
карточку Павла Николаева v8, 1 756 материалов миграционной приёмки, SQLite FK,
публичный snapshot и внешние HTTP-ответы. MODX, Dyn, Nginx, DNS и маршрут
`8443 → 443` скрипт не изменяет.

## Docker

```powershell
Copy-Item .env.example .env
# замените CMS_BOOTSTRAP_PASSWORD в .env
docker compose up -d --build
```

Порт 8000 доступен только на `127.0.0.1`; наружу сайт следует публиковать через HTTPS reverse proxy. Каталог `data` содержит SQLite и загруженные медиа и должен входить в резервное копирование.

## Состояние миграции

Read-only обход завершён: обследовано 1615 URL, из них 1606 рабочих и 9 возвращают 404. В новую CMS импортировано 1757 записей-черновиков и подготовлено 1606 редиректов. Из 13947 известных медиафайлов 13746 зеркалированы локально; 201 файл отсутствует на исходном сервере и задокументирован в отдельном отчёте. Исходный MODX при обходе и переносе не изменялся.

Перед переключением домена остаётся редакторская приёмка всех 1757 черновиков. Система намеренно не публикует непроверенный импорт; актуальное состояние всех барьеров находится в `outputs/migration-readiness.md`.

Для приёмки откройте нужный тип материала в CMS, включите фильтр «Только требующие проверки», сравните черновик со ссылкой на старую публичную страницу и нажмите «Сохранить и отметить проверенным». Обычное сохранение не снимает метку проверки, а публикация непроверенного импорта заблокирована на уровне API.

У импортированных материалов доступно отдельное поле «Текст со старого сайта»: в нём редактор удаляет старое меню, футер и устаревшие сообщения, не затрагивая фотографии и документы. Старые индексные URL перенаправляются в соответствующие разделы, а 1 578 детальных URL — на собственные страницы материалов новой CMS.
