# Пошаговый план завершения сайта и CMS

## Цель

Превратить текущий прототип в полноценный управляемый сайт: без демонстрационных данных и неработающих кнопок, с чистыми URL, реальным редактором, медиатекой, заявками, редакционным workflow, SEO и безопасным переносом архива.

## Этапы реализации

### 1. Зафиксировать исходное состояние

- Создать контрольный git-тег перед переделкой.
- Сделать согласованную резервную копию `cms.sqlite3` и `data/media`.
- Зафиксировать текущие показатели: 1 757 материалов, 1 опубликован, 1 756 требуют проверки, 1 606 редиректов.
- Добавить автоматические миграции схемы БД с таблицей `schema_migrations`.
- Убрать случайно опубликованную новость 2014 года обратно в очередь проверки.

Критерий готовности: база восстанавливается из копии, миграции повторяемы и не затрагивают старый MODX.

### 2. Исправить модель публикации и версий

- Ввести состояния `draft`, `in_review`, `scheduled`, `published`, `archived`, `trash`.
- Хранить номер опубликованной ревизии отдельно от редактируемой: изменение опубликованной страницы не должно сразу снимать её с сайта.
- Публикация фиксирует выбранную ревизию; восстановление версии создаёт новую ревизию.
- Добавить `scheduled_at`, `published_version`, `reviewed_by`, `reviewed_at`, `deleted_at`.
- Реализовать фоновую проверку отложенных публикаций раз в минуту.
- Сохранять действия редакторов в `audit_events`.

Критерий готовности: опубликованный материал остаётся видимым во время редактирования следующей версии; работают архив, корзина, восстановление и отложенная публикация.

### 3. Перевести публичный сайт на чистые URL

- Добавить Jinja2 и серверный рендеринг FastAPI.
- Заменить `#/...` на маршруты:
  - `/`
  - `/schedule`
  - `/about`
  - `/parish` и `/parish/{slug}`
  - `/school`
  - `/news` и `/news/{slug}`
  - `/gallery` и `/gallery/{slug}`
  - `/leaflet`
  - `/media`
  - `/pages/{slug}`
- Перенести существующий дизайн без визуального отката.
- Оставить JavaScript только для меню, диалогов, фильтров, lightbox и форм.
- Преобразовать существующие 301-редиректы с hash-адресов на чистые URL.
- Использовать `PUBLIC_BASE_URL=https://temple.gbvolkoff.name:8443`.

Критерий готовности: каждая страница открывается по самостоятельному URL, работает без hash-маршрутизации и получает корректный HTTP 404 при отсутствии материала.

### 4. Подключить все публичные разделы к CMS

- Главная получает из CMS активное «Главное», ближайшую будущую службу, последние новости, направления прихода, листок и контакты.
- При отсутствии данных показывать честное пустое состояние, а не вымышленные службы и даты.
- Расписание строить из опубликованных служб, группировать по датам, скрывать прошедшие и поддерживать PDF.
- «О храме» собирать из страниц, духовенства, святынь и контактов.
- «Жизнь прихода» выводить из `parish_section`; каждая карточка получает рабочую страницу.
- Воскресную школу сделать редактируемой страницей с расписанием и связанными материалами.
- Новости выводить с полным телом статьи.
- Галерею и листок загружать из CMS с пагинацией и всеми годами.
- Убрать все демонстрационные массивы, `data-toast-message` и прототипные тексты.
- Исправить ссылку на CMS в футере: вести на `https://cms.temple.gbvolkoff.name:8443/` либо убрать её для обычных посетителей.

Критерий готовности: на публичном сайте нет статичных дат, фальшивых материалов и кнопок-заглушек.

### 5. Реализовать полноценный редактор контента

- Заменить отдельные вручную описанные формы общей схемой типов материалов.
- Добавить структурированный блочный редактор без произвольного HTML:
  - абзац;
  - заголовок H2/H3;
  - изображение;
  - галерея;
  - цитата;
  - видео;
  - файл;
  - информационная плашка.
- Для блоков реализовать добавление, удаление, копирование и изменение порядка.
- Добавить полный текст для новостей, страниц, направлений и биографий.
- Реализовать связи между новостями, направлениями, галереями и страницами.
- Добавить редактирование подписей, alt-текстов и порядка фотографий.
- Предпросмотр строить тем же серверным шаблоном, что и публичную страницу.
- Убрать из production подсказку `admin / temple-demo`.

Критерий готовности: редактор может создать с нуля полноценную статью или страницу и увидеть точный внешний вид до публикации.

### 6. Реализовать медиатеку

- Добавить API списка файлов с поиском, фильтрами, пагинацией и информацией об использовании.
- Проиндексировать уже перенесённые файлы из `data/media`, не меняя существующие URL.
- Для изображений создавать миниатюры и WebP-варианты через Pillow.
- Проверять реальный тип файла, размер и расширение.
- Добавить выбор существующего файла из редактора.
- Реализовать изменение alt-текста, замену файла и безопасное удаление только неиспользуемых файлов.
- Для PDF показывать имя, размер, обложку и связанные материалы.
- Отдельно отметить 201 файл, отсутствовавший на старом сервере.

Критерий готовности: один файл можно загрузить один раз и повторно использовать в нескольких материалах; битые ссылки выявляются до публикации.

### 7. Доделать пользователей и редакционный workflow

- Оставить роли `viewer`, `editor`, `publisher`, `admin`.
- Добавить администратору создание, блокировку и изменение ролей пользователей.
- Добавить смену собственного пароля и принудительное завершение сессий.
- Добавить кнопку выхода.
- Ограничить кнопки интерфейса в соответствии с ролью.
- Реализовать действия:
  - сохранить черновик;
  - отправить на проверку;
  - вернуть редактору;
  - опубликовать;
  - запланировать;
  - архивировать;
  - переместить в корзину;
  - восстановить;
  - восстановить ревизию.
- Добавить массовую проверку, архивирование и публикацию выбранных материалов.

Критерий готовности: редактор не может самостоятельно публиковать, publisher не управляет пользователями, все действия отражаются в журнале.

### 8. Реализовать формы посетителей

- Создать таблицы `submissions` и `notification_outbox`.
- «Подать записку» сохраняет тип поминовения и имена; онлайн-оплата не входит в реализацию.
- «Записать ребёнка» сохраняет имя родителя, контакт, сведения о ребёнке и согласие на обработку данных.
- В CMS добавить очередь заявок со статусами `new`, `in_progress`, `done`, `spam`.
- Сначала сохранять заявку в БД, затем отправлять email-уведомление; ошибка почты не должна терять заявку.
- Настроить SMTP через переменные окружения, адрес получателя — через `SUBMISSION_NOTIFY_TO`.
- Добавить honeypot, ограничение частоты по IP и серверную валидацию.
- Автоматически удалять закрытые записки через 30 дней, заявки в школу — через 180 дней.

Критерий готовности: отправитель получает номер заявки, заявка появляется в CMS, повторная отправка и спам ограничиваются.

### 9. Реализовать поиск и SEO

- Создать SQLite FTS5-индекс по заголовкам, описаниям и текстовым блокам опубликованных материалов.
- Добавить `/search?q=` и API поиска с пагинацией.
- Генерировать для каждой страницы `title`, description, canonical, Open Graph и social preview.
- Добавить `sitemap.xml`, `robots.txt`, RSS новостей и структурированные данные организации/храма.
- Canonical URL временно включает внешний порт `:8443`.
- Добавить человекочитаемые страницы 404 и 500.
- Проверить все старые URL на отсутствие циклов и цепочек редиректов.

Критерий готовности: поиск находит архивные материалы; каждая публикация имеет отдельное социальное превью и присутствует в sitemap.

### 10. Провести редакторскую приёмку миграции

- Автоматически проверить обязательные поля, даты, кодировку, локальные медиа, дубли, старое меню и служебный HTML.
- Ввести ошибки трёх уровней: блокирующая, предупреждение, информационная.
- Вручную проверить в первую очередь:
  - контакты и реквизиты;
  - «Главное»;
  - актуальное расписание;
  - страницы «О храме»;
  - духовенство;
  - действующие направления;
  - последние новости и выпуски листка.
- Остальной архив принимать пакетами по типу и году, по 50 материалов.
- Материал с отсутствующей обязательной обложкой или PDF нельзя публиковать.
- Для необязательного отсутствующего изображения разрешить публикацию только после подтверждённого удаления или замены.
- После каждой партии выполнять автоматическую проверку ссылок и выборочно просматривать не менее 10% материалов.

Критерий готовности: `migration_review_required = 0`, нет публикаций с блокирующими ошибками, количество опубликованных материалов совпадает с утверждёнными партиями.

### 11. Резервирование, мониторинг и развёртывание

- Настроить ежедневный online-backup SQLite и инкрементальное копирование `data/media`.
- Хранить 7 ежедневных и 4 еженедельных копии; раз в месяц проверять восстановление.
- Добавить readiness-проверку БД и доступности каталога медиа.
- Логировать ошибки приложения, публикаций, email и планировщика.
- Проверять доступность:
  - `https://temple.gbvolkoff.name:8443/`
  - `https://cms.temple.gbvolkoff.name:8443/`
  - `/api/health`
- Развёртывать через новый Docker image после резервной копии и миграции БД.
- Nginx продолжает принимать внутренний `443`; внешний VPN/NAT остаётся `8443 → 443`.

Критерий готовности: контейнер можно обновить и откатить без потери БД, файлов и опубликованного состояния.

## Основные интерфейсы

- Публичные: серверные страницы, `/api/public/search`, `/api/public/submissions`.
- CMS: API материалов, workflow, ревизий, медиатеки, пользователей, заявок, настроек и редиректов.
- Все списки возвращают `{items, total, limit, offset}`.
- Блоки хранятся как JSON-массив `{id, type, data}`; произвольный HTML не принимается.
- Старый публичный API временно сохраняется для совместимости, затем удаляется после перехода серверных шаблонов.

## Проверка реализации

- Unit/API: роли, CSRF, версии, конфликты, расписание публикаций, корзина, формы, поиск, загрузка файлов.
- Миграция: повторный запуск не создаёт дублей и не перезаписывает редакторские изменения.
- Browser E2E: вся навигация, меню, закрытие диалогов, поиск, формы, публикация и восстановление версии.
- Адаптивность: 1600, 1024, 768, 390 и 320 px.
- Доступность: клавиатура, focus trap, Escape, подписи полей, alt-тексты и контраст.
- SEO: корректные canonical/OG/sitemap/RSS и отсутствие hash-URL.
- Финальный grep не должен находить `Прототип`, `будет подключено`, `data-toast-message` и демонстрационные даты.
- Все внутренние ссылки должны возвращать 200/301; CMS-файлы на публичном домене — 404.

## Принятые допущения

- Публичный сайт переводится на чистые серверные URL.
- Формы сохраняют заявки в CMS и отправляют email-уведомления.
- Архив принимается пакетами с автоматической проверкой и выборочным ручным контролем.
- Старый MODX и его админ-панель остаются строго read-only.
- Dyn не используется и не открывается.
- Текущая схема `внешний 8443 → внутренний 443` сохраняется.
