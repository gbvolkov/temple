# Текущие недоработки CMS

Дата фиксации: 21 июля 2026 года  
Источник: полный Playwright-аудит, постоянная регрессия Playwright (41 тест, 312 явных проверок), 80 серверных pytest-тестов и анализ текущей реализации

Все функциональные группы исследовательского аудита автоматизированы: `tests/e2e/coverage.json` не содержит `known_gaps`. На этапах 2–4 исправлены все зарегистрированные функциональные недостатки `CMS-01`–`CMS-17`; они выполняются как положительная регрессия. Открытых функциональных `CMS-*` дефектов нет. Source-data профиль отдельно контролирует три расхождения данных `GAP-*`, которые требуют следующих этапов миграции.

## Приоритеты

- **P0** — блокирует production cutover;
- **P1** — высокий риск потери данных, доступа или ошибочного редакторского решения;
- **P2** — существенный функциональный/CJM/UX-дефект;
- **P3** — улучшение качества, ясности или сопровождения.

## Подробности

### GAP-01 — миграционная приёмка не завершена

**Приоритет:** P0, release blocker.

Исходное состояние проекта: 1 757 импортированных материалов требуют редакторской приёмки. Полный audit на тестовой копии до QA-финализации показал:

- 1 470 блокирующих проблем;
- 3 048 предупреждений;
- 1 757 информационных отметок.

Playwright подтвердил, что механизм партий работает, но тестовая финализация 41 объекта выполнялась только в изолированной копии и не меняла рабочую БД.

**Риск:** непроверенный архив нельзя безопасно публиковать или использовать для cutover.

**Что сделать:** запускать audit на актуальной рабочей копии, исправлять blockers, принимать приоритетные разделы, затем архив партиями по типу/году. Cutover разрешать только при `migration_review_required = 0` для утверждённого объёма и отсутствии blocker у публикуемых версий.

### GAP-02 — реестр потерянных медиа не связан с панелью

**Приоритет:** P0 до выяснения.

README и миграционные отчёты фиксируют 201 файл, отсутствовавший на старом сервере. При этом вкладка **«Утрачено»** в копии текущей БД вернула `0 утраченных файлов`, поэтому conditional-кнопку **«Загрузить замену»** невозможно было проверить на реальной записи.

**Риск:** редактор получает ложный сигнал, что потерянных файлов нет; публикации могут содержать битые ссылки.

**Что сделать:** сверить `outputs/missing-legacy-media.*`, manifest и таблицы `missing_media_issues*`; добавить проверку количества в rollout; заполнить панель или явно пометить issues как разрешённые с доказательством.

## Исправлено на этапе 2

Следующие записи сохранены как история дефектов и имеют `status: fixed` в `tests/e2e/defects.json`.

### CMS-01 — тихая потеря несохранённых изменений

**Приоритет:** P1.

**Шаги:** открыть новый/существующий материал, изменить поле, не сохранять, нажать другой тип, другую запись или «Новый материал».

**Факт:** переход выполняется без confirm, введённое значение пропадает.

**Ожидание:** модальный выбор «Сохранить / Не сохранять / Остаться»; защита также для Back/refresh/закрытия вкладки через `beforeunload`.

**Критерий приёмки:** ни один переход из `dirty=true` не уничтожает ввод без явного решения пользователя.

### CMS-02 — пустые блоки ломают preview

**Приоритет:** P1.

Добавление изображения, галереи, видео, файла и ряда других пустых блоков немедленно отправляет невалидный payload. Сервер отвечает 422, preview показывает технический JSON `{"detail":"Блок …"}`, консоль засоряется повторными ошибками.

**Что сделать:** не отправлять preview для локально неполного блока; показывать inline-ошибку возле карточки; сохранять последний успешный preview; не выводить raw JSON.

**Критерий:** добавление пустого блока не создаёт console error и не уничтожает предыдущий preview.

### CMS-03 — рассинхронизация фильтра chooser медиатеки

**Приоритет:** P1.

После выбора `usage=used`, закрытия и повторного открытия chooser UI показывает **«Любое»**, но запрос остаётся `/api/admin/media?...&usage=used`. В результате отображается «Файлы не найдены» при наличии подходящего файла.

Причина в `openMediaChooser`: DOM-значения очищаются, но `state.media.q` и `state.media.usage` не сбрасываются.

**Что сделать:** атомарно сбрасывать DOM и state или сохранять оба значения намеренно; добавить E2E на close/reopen.

### CMS-04 — отозванная сессия не возвращает вход

**Приоритет:** P1.

После **«Завершить сессии»** следующий запрос пользователя получает 401 и toast «Сессия истекла», но UI продолжает выглядеть авторизованным, login-dialog не открывается.

**Риск:** пользователь не понимает, почему действия не работают; несохранённый текст может быть потерян.

**Что сделать:** централизованно обрабатывать 401 в `apiRequest`: сохранить/предупредить о dirty-state, очистить session UI, открыть вход, после входа повторно загрузить контекст.

### CMS-05 — мобильный пользователь не может сменить пароль или выйти

**Приоритет:** P1.

CSS скрывает `.cms-topbar__actions .button--ghost` на `max-width:760px`. Это единственные кнопки **«Пароль»** и **«Выйти»**; в мобильном меню дублей нет.

**Что сделать:** добавить account-section в sidebar или отдельное меню профиля. Проверить 390 и 320 px.

### CMS-06 — необработанная ошибка пилотной партии

**Приоритет:** P1.

Кнопка **«Создать пилотную партию»** активна до наличия завершённого полного аудита. Сервер отвечает 409 «нет актуального завершённого аудита», обработчик клика не имеет `.catch`, пользователь не получает toast, в браузере — `pageerror`.

**Что сделать:** disabled-state с пояснением до completed audit; обернуть действие в обработку ошибок; предложить кнопку запуска аудита и показывать его статус.

## Исправлено на этапе 3

### CMS-07 — несогласованное состояние после «Новый материал»

**Приоритет:** P2.

Форма становится новой, но поиск остаётся «Священство», checkbox проверки включён, select продолжает показывать ID старой записи.

**Риск:** редактор может решить, что меняет существующую запись, либо наоборот.

**Исправление:** поиск, фильтр редакционной проверки, выбранная запись, список и счётчик теперь сбрасываются согласованно; новый запрос списка не содержит старых параметров.

### CMS-08 — login-dialog нельзя открыть повторно

**Приоритет:** P2.

После крестика/Отмены вход закрыт, а публичной кнопки «Войти» в CMS нет. Единственный обход — reload.

**Исправление:** в topbar для гостя отображается постоянная кнопка «Войти»; повторное открытие переводит фокус в имя пользователя.

### CMS-09 — невалидный username `pattern`

**Приоритет:** P2.

Шаблон `[A-Za-z0-9][A-Za-z0-9._-]{2,79}` становится невалидным в современном `pattern` с флагом `v`: дефис внутри класса нужно экранировать или поставить первым. Chromium пишет console error и считает `a!b` валидным; только сервер затем отклоняет submit.

**Исправление:** дефис экранирован; Chromium отклоняет `a!b` и принимает допустимое `a-b` без console error.

### CMS-10 — skip-link не завершает перевод фокуса

**Приоритет:** P2, accessibility.

Ссылка первая в tab-order и меняет URL на `#editor`, но active element после Enter — `BODY`. У `<main id="editor">` нет `tabindex="-1"`.

**Исправление:** `main#editor` получил `tabindex="-1"`, а обработчик skip-link явно переводит фокус и прокручивает основную область.

## Исправлено на этапе 4

### CMS-11 — поиск миграционных проблем не ищет код

**Приоритет:** P2.

Placeholder обещает **«Код, заголовок или старый URL»**. Поиск `required_field_missing` вернул 0, хотя в таблице было 879 открытых issues с этим кодом.

Backend `q` ищет `title`, `legacy_url`, `message`, но не `i.code`.

**Исправление:** backend ищет `q` также по `i.code`; точный код возвращает соответствующие проблемы, а API- и Playwright-тесты сверяют код и уровень.

### CMS-12 — race при обновлении migration dashboard

**Приоритет:** P2.

`refreshMigrationStatus()` запускает параллельно status/issues/batches, но не отменяет предыдущий refresh и не проверяет sequence. При быстром изменении нескольких фильтров более поздний ответ старого запроса может перерисовать выдачу после нового.

**Исправление:** предыдущий refresh отменяется через `AbortController`, sequence ID запрещает устаревшему ответу перерисовать экран, а фильтр получает согласованные busy/disabled-состояния. E2E искусственно переставляет ответы и подтверждает latest-only rendering.

## Исправлено на этапе 3 — продолжение

### CMS-13 — неоднозначная семантика обязательных полей

**Приоритет:** P2.

Поле обложки помечено `*`, но неполный draft успешно сохраняется. Сервер корректно блокирует `submit-review/publish` через `require_ready`, однако UI не объясняет, что обязательность относится к готовности, а не к сохранению черновика.

**Исправление:** форма объясняет семантику `*`; перед review, schedule и publish показывает сводку недостающих полей, ставит `aria-invalid` и фокусирует первое поле без отправки невалидного workflow-запроса.

### CMS-14 — «Настройки» без настроек

**Приоритет:** P2/P3 в зависимости от требований владельца.

Панель показывает только «Контентная схема 1.6.0». Нет параметров сайта, branding, уведомлений, контактов окружения или диагностического статуса.

**Исправление:** панель стала read-only диагностикой приложения, схемы, SQLite, FTS-индекса, медиа-хранилища и миграции с ручным обновлением. API требует сессию и не возвращает секреты или filesystem paths.

## Исправлено на этапе 4 — продолжение

### CMS-15 — долгий reindex без прогресса

**Приоритет:** P3.

13 701 файл обработан за 49,4 секунды. Пользователь видит только «Проверяем…», без процента, количества и возможности понять, зависла ли операция.

**Исправление:** reindex выполняется долговечной фоновой задачей в SQLite с фазами, числовым прогрессом, счётчиками и безопасным журналом ошибок. После перезагрузки UI продолжает polling, а второй одновременный запуск получает `409` с ID активной задачи.

### CMS-16 — большая выдача selector неочевидна

**Приоритет:** P3.

В новостях selector показал 100 из 733. Поиск работает, но нет явного текста, что остальные записи доступны через поиск, и нет пагинации списка.

**Исправление:** selector показывает диапазон и общее число, поддерживает страницы по 100 записей, кнопки «Назад»/«Вперёд», объясняет лимит страницы и сбрасывает offset при поиске и смене фильтра.

### CMS-17 — deprecated тестовый клиент

**Приоритет:** P3.

Ранее все 75 тестов проходили с `StarletteDeprecationWarning`: использование `httpx` через `starlette.testclient` было deprecated.

**Исправление:** тестовая зависимость переведена на совместимый `httpx2`; 80 серверных тестов проходят, а отдельный subprocess-тест запускает TestClient с warnings-as-errors.

### GAP-03 — расхождение числа медиа

**Приоритет:** P3 до анализа.

Playwright reindex сообщил 13 701 файл, а проектный отчёт — 13 746 зеркалированных. Разница 45 может быть объяснима неподдерживаемыми/служебными объектами, но сейчас объяснение не выведено.

**Что сделать:** отчёт `indexed / skipped / unsupported / duplicate / missing` и rollout-assert для ожидаемых категорий.

## Рекомендуемый порядок исправления

1. GAP-02 и GAP-03 — сверка реального списка и количества медиа (этап 5).
2. Автоматически подготовить однозначные исправления материалов (этап 6).
3. Завершить GAP-01 редакционными партиями после автоматической подготовки (этапы 7–8).

## Минимальные release-критерии

- нет тихой потери dirty-форм;
- 401 всегда приводит к понятному повторному входу;
- account actions доступны на mobile;
- preview не показывает raw JSON;
- chooser state и UI совпадают;
- migration audit/pilot имеют управляемый status/error flow;
- список missing media согласован с миграционным отчётом;
- у публикуемых материалов нет blocker и незаполненных required;
- backup/restore drill и production smoke пройдены.
