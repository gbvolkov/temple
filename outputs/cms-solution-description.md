# Полное описание решения

## 1. Назначение

Решение заменяет старую публичную часть MODX новым серверным сайтом и отдельной CMS. Старый сайт используется только как read-only источник миграции. Новые черновики, публикации, файлы, пользователи и заявки хранятся в новой системе.

Состав решения:

- публичный адаптивный сайт;
- схема-управляемый редактор CMS;
- редакционный workflow и история ревизий;
- медиатека с повторным использованием файлов;
- формы посетителей и очередь уведомлений;
- роли и управление сессиями;
- инструменты полного импорта и миграционной приёмки;
- поиск, SEO, sitemap, RSS и чистые публичные URL.

## 2. Архитектура

| Слой | Реализация | Ответственность |
|---|---|---|
| HTTP-приложение | FastAPI | API, авторизация, workflow, публичные маршруты, фоновые задачи |
| Публичный render | Jinja2 | SSR-страницы с настоящими 200/404 и чистыми URL |
| CMS UI | HTML/CSS/vanilla JavaScript | Формы, блоки, preview, панели управления |
| Контентная схема | `site/cms-schema.json`, версия 1.6.0 | Типы, поля, блоки, порядок навигации |
| Хранилище | SQLite | Контент, ревизии, пользователи, заявки, медиа, аудит, редиректы |
| Файлы | `data/media` и `data/media-derivatives` | Оригиналы, thumbnails и WebP |
| Фоновые процессы | asyncio scheduler | Отложенные публикации, email-outbox, migration audit |
| Миграция | Python-модули `server/*import*`, `legacy_*`, `migration_*` | Crawl, импорт, зеркалирование, аудит и приёмка |

Приложение запускается одним FastAPI-процессом. Публичный сайт и CMS используют одну БД, поэтому опубликованная ревизия становится видимой без отдельной синхронизации.

## 3. Публичные маршруты

Основные SSR-маршруты:

- `/` — главная;
- `/schedule` — богослужения;
- `/about` — о храме;
- `/about/clergy/{slug}` — духовенство;
- `/parish` и `/parish/{slug}` — жизнь прихода;
- `/school` — воскресная школа;
- `/news` и `/news/{slug}` — новости;
- `/gallery` и `/gallery/{slug}` — фотоальбомы;
- `/leaflet` — выпуски листка;
- `/media` — видео и архив;
- `/pages/{slug}` — универсальные страницы;
- `/search` — поиск;
- `/sitemap.xml`, `/robots.txt`, `/rss.xml` — SEO/индексация.

Публичная detail-страница читает только `published_version` и `published_slug`. Рабочая версия может изменяться независимо и не попадает на сайт до следующей публикации.

## 4. Типы контента

CMS строит формы из схемы, а не из отдельных вручную написанных экранов.

| Тип | Назначение | Основное размещение |
|---|---|---|
| `home_feature` | Главные промо-блоки и содержание главной | `/` |
| `news` | Новости и анонсы | `/news/{slug}` |
| `service` | События расписания богослужений | `/schedule` |
| `gallery` | Фотоальбомы | `/gallery/{slug}` |
| `leaflet_issue` | Выпуски «Иннокентиевского листка» | `/leaflet` |
| `parish_section` | Направления приходской жизни | `/parish/{slug}` |
| `page` | Универсальные страницы | `/pages/{slug}` или тематический маршрут |
| `clergy` | Карточки духовенства | `/about/clergy/{slug}` |
| `video` | Видео и трансляции | `/media` |
| `site_contact` | Адрес, телефон, email и социальные сети | `/about#contacts`, футер |

Общие возможности полей:

- строка, многострочный текст, дата, время, число;
- select и checkbox;
- одиночное изображение/файл;
- список изображений;
- расписание из строк;
- список социальных ссылок;
- связи с другими материалами;
- SEO title, description и social image.

## 5. Блочная модель

Текстовое содержание хранится структурированным JSON-массивом блоков `{id, type, data}`. Произвольный HTML не принимается.

Поддерживаются:

1. `paragraph` — абзац с bold, italic и безопасными ссылками;
2. `heading` — структурированный заголовок;
3. `list` — маркированный или нумерованный список;
4. `image` — изображение, alt и подпись;
5. `gallery` — список изображений;
6. `quote` — цитата;
7. `video` — видео/трансляция;
8. `file` — документ для скачивания;
9. `callout` — информационная плашка.

Блок можно копировать, удалять и перемещать. Перед preview клиент проверяет обязательные данные image, gallery, video и file, показывает inline-ошибки и сохраняет последний успешный iframe. Валидный preview генерируется сервером тем же шаблонным слоем, что и публичная страница.

## 6. Модель контента и ревизий

Главные сущности:

- `contents` — текущее рабочее состояние и указатель на опубликованную ревизию;
- `revisions` — неизменяемые снимки версий;
- `audit_events` — журнал действий;
- `redirects` — старые пути и целевые чистые URL;
- FTS5-таблицы `content_search*` — полнотекстовый индекс.

Важные поля материала:

- `status`;
- `version` — версия рабочего состояния;
- `published_version` — версия, видимая посетителям;
- `slug` и `published_slug`;
- `scheduled_at`, `published_at`, `reviewed_at`, `deleted_at`;
- `migration_review_required`;
- `data_json` — значения схемы.

Каждая mutation передаёт ожидаемую `version`. Если другая сессия уже изменила объект, сервер возвращает 409 вместо тихого перезаписывания.

## 7. Редакционный workflow

```text
draft → in_review → published
  ↑        │          │
  │        ├→ scheduled ─→ published
  └────────┘          │
        restore ← archived/trash
```

Правила:

- редактор создаёт и сохраняет рабочую версию;
- редактор отправляет материал на проверку;
- publisher/admin возвращает, публикует сразу или назначает дату;
- scheduler публикует наступившие `scheduled`-материалы;
- архив и корзина скрывают публичную страницу;
- восстановление создаёт снова скрытый черновик;
- восстановление ревизии копирует старые данные в новую рабочую версию;
- опубликованная версия остаётся доступна, пока новая рабочая версия редактируется.

Редакционная очередь выполняет массовую публикацию и архивирование атомарно: при конфликте не меняется ни один выбранный объект.

Клиент различает сохранение черновика и готовность к смене статуса. Неполный черновик разрешено сохранять, но перед `submit-review`, `schedule` и `publish` интерфейс вычисляет обязательные поля по схеме, показывает сводку, назначает `aria-invalid` и переводит фокус к первому пропуску. Невалидный workflow-запрос на сервер не отправляется; серверная `require_ready` остаётся вторым уровнем защиты.

Dirty-состояние редактора защищено единым navigation guard с вариантами сохранить, отбросить или остаться; для закрытия и перезагрузки используется `beforeunload`. При истечении сессии DOM редактора сохраняется, защищённые действия блокируются, а после повторного входа пользователь продолжает работу с теми же значениями.

## 8. Роли и безопасность доступа

Роли образуют возрастающую иерархию:

| Роль | Права |
|---|---|
| `viewer` | Просмотр материалов, истории, медиатеки и migration dashboard |
| `editor` | Viewer + создание/редактирование, загрузка медиа, отправка на проверку, решения по draft-партии |
| `publisher` | Editor + публикация, расписание, архив/корзина, массовые действия, заявки, аудит и финализация миграции |
| `admin` | Publisher + пользователи, завершение сессий, reindex и удаление неиспользуемых файлов |

Сессии хранятся сервером и используют CSRF-token для mutation. Смена пароля завершает все сессии пользователя. Администратор может отдельно отозвать чужие сессии, заблокировать учётную запись или изменить роль.

Development-пароль `temple-demo` активен только при `CMS_ENV=development`. В production должен быть задан длинный `CMS_BOOTSTRAP_PASSWORD` до первого запуска.

## 9. Медиатека

Таблица `media` хранит метаданные оригинала, checksum, MIME, размер, тип, alt, версию и источник. `media_usages` связывает файл с рабочими и историческими версиями материалов.

Возможности:

- поиск по имени, alt и использующему материалу;
- фильтры image/video/document и used/unused;
- пагинация по 48 объектов;
- повторное использование без создания копии;
- постоянный URL оригинала;
- thumbnails/WebP для изображений;
- изменение alt;
- replacement как новый media object вместо перезаписи старого;
- удаление только неиспользуемого файла администратором;
- долговечный фоновый reindex существующего каталога с фазами, прогрессом, счётчиками и журналом ошибок;
- отдельный список потерянных legacy-файлов.

Ограничения размеров и допустимые расширения задаются переменными `CMS_MAX_*_BYTES` и серверной валидацией реального типа файла.

Задачи reindex хранятся в `media_reindex_jobs`. В один момент времени разрешена одна активная задача; после перезапуска незавершённая задача возвращается в очередь. `POST /api/admin/media/reindex` отвечает `202`, а состояние доступно через `GET /api/admin/media-reindex` и `GET /api/admin/media-reindex/{job_id}`. Применение найденных изменений выполняется одной SQLite-транзакцией; пути файловой системы и секреты в ошибках API редактируются.

## 10. Заявки посетителей

Поддерживаются два типа:

- `prayer_note` — вид поминовения и 1–10 имён;
- `school_enrollment` — родитель, контакт, ребёнок, возраст, комментарий и согласие.

Сначала заявка записывается в SQLite и получает reference code. Затем `notification_outbox` независимо пытается отправить email. Поэтому ошибка SMTP не теряет обращение.

Статусы обработки: `new`, `in_progress`, `done`, `spam`. История переходов записывается в `submission_events`. Закрытые записки очищаются через 30 дней, школьные обращения — через 180 дней.

Защита форм:

- серверная валидация;
- honeypot;
- rate limit по HMAC-хешу IP;
- исходный IP не сохраняется;
- доверенные proxy-сети задаются явно;
- персональные данные не входят в list-response, а открываются только в карточке.

## 11. Миграция и приёмка

Миграционный pipeline:

1. read-only crawl старого сайта;
2. инвентаризация URL и медиа;
3. зеркалирование файлов с SHA-256 и checkpoint;
4. идемпотентный импорт черновиков;
5. создание 301-редиректов;
6. автоматический audit;
7. редакционные партии;
8. атомарная финализация решений.

Audit проверяет обязательные поля, даты, кодировку, HTML/служебный текст, media, alt, внутренние и внешние ссылки, дубли и provenance. Уровни: blocker, warning, info.

Выдача проблем ищет по коду, заголовку, старому URL и сообщению. Клиент отменяет устаревшие запросы dashboard и применяет только последний ответ, поэтому быстрое переключение фильтров не может вернуть интерфейс к старому состоянию.

Партия фиксирует конкретные версии материалов и результатов аудита. Для каждого объекта редактор выбирает `accept`, `archive` или `trash`, отмечает ручной просмотр и подтверждает индивидуальные warnings. Publisher подтверждает общие warnings и финализирует атомарно. При конфликте версии вся операция останавливается.

Финализация миграции не публикует материалы: она снимает migration-флаг или переводит объект в выбранное непубличное состояние. Публикация остаётся отдельным workflow.

## 12. Поиск и SEO

Поиск использует SQLite FTS5 по заголовкам, описаниям и текстовым блокам опубликованных ревизий. Публичные страницы получают:

- уникальный `<title>` и description;
- canonical URL;
- Open Graph;
- social preview;
- JSON-LD;
- sitemap и RSS.

Старые hash-адреса поддерживает одноразовый compatibility-мост. Новая навигация создаёт только чистые URL.

## 13. Конфигурация и эксплуатация

Авторизованная read-only точка `GET /api/admin/diagnostics` агрегирует версию приложения и схемы, SQLite `quick_check` и foreign keys, состояние FTS, согласованность записей и оригиналов медиа, а также состояние импорта и миграционного аудита. Ответ намеренно не содержит секретов и filesystem paths. Панель «Настройки и диагностика» визуализирует статусы и поддерживает ручное обновление.

Ключевые переменные:

- `CMS_ENV`;
- `CMS_DATABASE`;
- `CMS_MEDIA_DIR`;
- `CMS_MEDIA_DERIVATIVES_DIR`;
- `CMS_BOOTSTRAP_USER`, `CMS_BOOTSTRAP_PASSWORD`;
- `PUBLIC_BASE_URL`;
- `SMTP_*`, `SUBMISSION_NOTIFY_TO`;
- `SUBMISSION_IP_HASH_SECRET`;
- `SUBMISSION_TRUSTED_PROXY_NETWORKS`.

Локальный запуск:

```powershell
uv sync
$env:CMS_ENV="development"
uv run uvicorn server.app:app --host 127.0.0.1 --port 4174
```

Проверки:

```powershell
uv run pytest -q
node --check site/app.js
node --check site/cms.js
```

Production deployment должен включать согласованный SQLite backup, копию `data/media`, проверку восстановления, миграции схемы, readiness и smoke-тест. Порт приложения предназначен для loopback; наружу сайт публикуется через HTTPS reverse proxy.

## 14. Текущая граница готовности

Техническая платформа и ключевые workflow существуют и проверены. До production cutover требуется:

- исправить дефекты из `outputs/cms-current-shortcomings.md`;
- завершить редакционную приёмку мигрированных материалов;
- устранить blockers аудита;
- настроить production secrets, SMTP, backup/restore и HTTPS;
- выполнить контрольный rollout на восстановленной копии.
